INTRODUCTION OF EXCEPTION HANDLING

Exception handling is the process of responding to the occurrence, during computation, of exceptions. Exception handling is one of the most common ways of dealing with errors within a program. Error handling is a big issue for programming languages. What do you do when a program doesn't act according to plan? Different languages have different answers to this problem.

DIFFERENT METHODS OF ERROR HANDLING

RETURN VALUES

Another common method of handling errors is to output whether a function was successful or not within the return value of the function. This is used in many lower-level languages such as C, C++, and Rust. Typically functional languages use this paradigm of error handling because it does not break the flow of the program. The newer langages that use this kind of error handling usually use a coupled structure as the return value that return the value, if available, and an error if there was one.

Rust uses the Option data structure to declare that there might not be a result in some degenerative cases:

fn file_path_ext_explicit(file_path: &str) -> Option<&str> { match file_name(file_path) { None => None, // No result, so return None Some(name) => match extension(name) { // Normal expected result None => None, Some(ext) => Some(ext), } } }

There is also the Result data structure used to declare that there might be an error in some degenerative cases:

fn double_number(number_str: &str) -> Result<i32, ParseIntError> { match number_str.parse::<i32>() { Ok(n) => Ok(2 * n), // Return normal expected result Err(err) => Err(err) // This line returns the result as an error } }

Many functions in languages like C will return a NULL value or 0 if there was an error during the function's execution. Type typical return values that indicate failure within a function's execution usually follow these rules:

For non-pointer return types

  • 0: success
  • -1: failure

For pointer return types

  • NULL: failure
  • Everything else: success

The non-pointer return types can cause some ambiguity when the return values is supposed to be used for more than just error handling. For example, the atoi function in C is used to parse a string of text to an integer. The return type of the function is an int, but a return value of 0 indicates that the function failed. What if the input was a string value that represented 0? How would you differentiate "0" vs an error ocurring within the function? atoi does not include support for such functionality. This is why out parameters are used in many cases in C.

OUT PARAMETERS

Out parameters are parameters that are passed to a function by reference and require a value be assigned to them before the function is returned.

This method is also common within C programs. Many functions have the following design pattern:

int my_function(char* success) { ... if (success) { *success = 1; } else { *success = 0; } ... }

With this design pattern, you have to be a little more verbose with how you call functions:

char successful; int returnValue = my_function(&successful); if (!successful) { // deal with error }

You are required to pass in the reference to a variable, which requires that you have a variable defined in the first place. While this addresses the ambiguity between return values vs errors, you pay the price in being verbose and the possible performance implications of having to allocate space for an extra variable used just for determining the success of a function.

GLOBAL VARIABLE

Global variables are often to indicate the latest error in a program. An example of this is the errno library in C which is defined in the errno.h file. The errno library uses a single global integer variable errno to deal with errors in a program. The out parameter code we used before can be modified to follow this design pattern:

int my_function() { ... if (success) { ... // do not set errno } else { // some integer that corresponds to type of error errno = 1; } ... }

Instead of taking in a success out parameter, the function sets the global variable to a specific value if an error occurs. If no error occurs, then the errno global variable is not set. The function can be called without having to define a variable:

int returnValue = my_function(); if (errno != 0) { // deal with error }

This cleans up the function call some, but just as the other methods, it has some drawbacks to it as well. With a global variable being used, you have to be more careful in multi-threaded programs that are calling functions that read/write to the errno variable. To keep errno thread-safe, you have to use a mutex or semaphore. Both of those will impact the performance of the program.

DESIGN BY CONTRACT

Design by Contract (DbC) handles errors by requiring correct input and correct output. A function does this by specifying pre-condition and post-condition states that the function must fulfill to be considered a success.

function myFunc(x: Float): Float pre-condition (x >= 0) post-condition (return >= 0) begin ... end

This method of error handling is often coupled with exception handling. If a user were to call this function in such a way that violates the pre or post condition, an exception would be thrown. However, in some cases a pre/post condition can be validated at compile time. In these cases, a compile-time error can be used to prevent the function from returning an exception.

Because DbC is often coupled with exception handling, it can be implemented in Nova. With annotations such as [PreCondition <Bool expression>] and [PostCondition <Bool expression>], DbC could have a place in Nova's future.

[PreCondition x >= 0] [PostCondition return >= 0] myFunc(Float x) -> Float { ... }

THROWN EXCEPTIONS

The standard Java/C# style of handling errors in code is to throw exceptions. This is done by defining an area in the code to catch exceptions, and an area to throw exceptions. A catch without any corresponding throws is useless. The common design pattern for try/catch/throw looks something like this:

myFunc() { try { sketchyFunction() } catch (Exception e) { // deal with the exception } } sketchyFunction() { if (someInvalidCondition) { throw new Exception("Bad thing happened") } else { // do whatever sketchyFunction is supposed to do } }

However, unlike Java, exceptions are not required to be caught in Nova, so the following code is also valid.

myFunc() { sketchyFunction() } sketchyFunction() { if (someInvalidCondition) { throw new Exception("Bad thing happened") } else { // do whatever sketchyFunction is supposed to do } }

This can be helpful for when you know the function won't have an error, or you just don't want to deal with any exceptions from it.

NOVA'S ERROR HANDLING

Nova uses exceptions to handle errors. The reason for this choice stems from several factors:

  • Return values, if not using a Result<Type, Error> or Optional<Type> type, can be ambiguous.
  • Return values, if using a Result<Type, Error> or Optional<Type> type, can be verbose and impact performance (e.g. unwrapping, packaging return value, etc).
  • Out parameters are not even supported in Nova because they almost always cause unnecessary complexities.
  • Global variables are not thread-safe and are generally accepted as bad practice.
  • Design by contract is compatible with exception handling.

TOSS

In addition to traditional try/catch/throw functionality, Nova also includes "toss" functionality. Tossing an exception is analogous to throwing an exception. The difference is that when you toss an exception, it only breaks the execution flow if there is a catch block waiting for that specific type of exception. For example, you can replace the throw keyword with toss:

myFunc() { sketchyFunction() } sketchyFunction() { if (someInvalidCondition) { toss new Exception("Bad thing happened") } else { // do whatever sketchyFunction is supposed to do } // OTHER CALCULATIONS AND STUFF }

The difference that this has on the execution flow is that instead of stopping at the toss new Exception("Bad thing happened") as it would have with throw new Exception("Bad thing happened"), the code will continue to execute. Now if we were to catch the exception, you would have a different execution flow:

myFunc() { try { sketchyFunction() } catch (Exception e) { // deal with exception } } sketchyFunction() { if (someInvalidCondition) { toss new Exception("Bad thing happened") } else { // do whatever sketchyFunction is supposed to do } // OTHER CALCULATIONS AND STUFF }

In this example, the toss would behave just as a throw because there is a catch block waiting for an exception.
